## Project Overview

Angular 22 frontend (PWA) for the CSI Madhya Kerala Diocese Premarital Counselling Centre. Public registration flows (premarital, abroad/outside-Kerala premarital, general, pre-confirmation) plus a JWT-protected admin dashboard. English/Malayalam i18n. Deployed to Azure Static Web Apps; talks to an ASP.NET Core backend (sibling repo `../csi-mkd-premarital-app-BE`) and an Azure Functions app.

## Common Commands

- `npm start` тАФ dev server on localhost:4200 (serve defaults to the **development** configuration тЖТ local APIs)
- `ng build` тАФ build (defaults to the **production** configuration тЖТ Azure API URLs, service worker on)
- `ng build --configuration=development` тАФ dev build (localhost APIs, source maps)
- `ng test` тАФ unit tests (Karma/Jasmine)
- `ng test --include='**/some-name.spec.ts'` тАФ run a single spec file
- `npm run gen-api` тАФ regenerate both API clients (see below); requires the backend(s) running locally since the OpenAPI specs are fetched from live URLs
- `npm run extract-i18n` тАФ extract i18n messages to `src/locale/`
- `./deploy.sh` тАФ bump patch version, production build, copy `en/index.html` to root, deploy via Azure SWA CLI

Requires Node >= 22, npm >= 10.

## Architecture

### Two backends, two generated API clients

`src/api/` is entirely auto-generated by `ng-openapi-gen` тАФ **never hand-edit it**:

- `src/api/api-main-app/` тАФ main ASP.NET Core API (`ng-openapi-gen.json`, spec from `localhost:5177/swagger/v1/swagger.json`)
- `src/api/api-functions/` тАФ Azure Functions app (`ng-openapi-gen-functions.json`, spec from `localhost:7071/api/swagger.json`)

API root URLs are **build-time constants** injected via the `define` option in `angular.json` (`API_ROOT_URL_MAIN_APP`, `API_ROOT_URL_FN_APP`, `APPLICATION_INSIGHTS_CONNECTION_STRING`; typed in `src/types.d.ts`). There are no `environment.ts` files. `app.config.ts` assigns them to each client's `ApiConfiguration` in an app initializer.

### App configuration (`src/app/app.config.ts`)

- **Zoneless change detection** (`provideZonelessChangeDetection()`)
- Router uses **hash location** (`withHashLocation()` тАФ all URLs are `/#/...`), view transitions, component input binding
- HTTP interceptors: `tokenInterceptor` (attaches JWT) and `rateLimitInterceptor`
- Global error handler (`GlobalErrorHandlerService`), Application Insights analytics services initialized at startup
- Service worker enabled only in production builds; `MAT_DATE_LOCALE` is `en-GB` (DD/MM/YYYY)

### Routing & layouts (`src/app/app.routes.ts`)

Everything is lazy-loaded standalone components under two shells:

- `PublicLayout` тАФ about, team, sessions, and the four registration flows in `src/app/register/`
- `AdminLayout` тАФ guarded by `authGuard`; list/management pages in `src/app/admin/` (premarital, abroad-premarital, general-list, pre-confirm-list, session-config, instructors, feedback/question answers, database-backup). `admin/login` sits outside the layout.

Registration and feedback forms use `pendingChangesGuard` (`canDeactivate`) to warn about unsaved changes.

### Key directories

- `src/app/core/` тАФ auth (guard/service/interceptor), guards, interceptors, validators (email domain, unique email, date range), and singleton services (church data, sessions fallback, blob access, file upload, certificates, theme, analytics, PDF/calendar helpers)
- `src/app/shared/` тАФ shared dialogs/components (certificate preview, archive-retrieval and rehydration dialogs, etc.)
- `src/locale/` тАФ `messages.xlf` (en source) and `messages.ml.xlf`; builds are localized (`localize: true`), so output is per-locale (`dist/csi-mkd-counselling/browser/{en,ml}/`)

### Admin edit dialogs тАФ known pitfalls

Submit buttons are typically `[disabled]="form.invalid"`. Two recurring traps that silently keep forms invalid:

- A control declared with `Validators.required` but no corresponding field in the template
- `[min]`/`[max]` on Material datepickers acting as validators тАФ never enforce a min of "today" when editing existing records whose dates are in the past

`FormArray.push/removeAt` does not mark the form dirty; call `markAsDirty()` explicitly if the UI depends on dirty state.

## Coding Conventions

- Standalone components only (do not set `standalone: true` тАФ it's the default); `ChangeDetectionStrategy.OnPush` (some files use `.Eager`, Angular's renamed `Default` тАФ prefer OnPush for new code)
- Signals for state (`signal`, `computed`, `toSignal`); `inject()` over constructor injection; `input()`/`output()` functions over decorators
- Native control flow (`@if`, `@for`, `@switch`); `class`/`style` bindings instead of `ngClass`/`ngStyle`; host bindings via the `host` object, not `@HostBinding`/`@HostListener`
- Reactive forms preferred; avoid `any` (lint warns); strict type checking
- **`no-console` is an ESLint error** тАФ use `console.error`/`console.warn` only for critical issues; user feedback goes through MatSnackBar (see `DEVELOPER_GUIDELINES.md`)
- Lint also enforces Prettier, warns on missing `data-testid` in templates, and warns on subscriptions without `takeUntil`-style teardown
- Styling: SCSS + Angular Material (azure-blue prebuilt theme) + Tailwind CSS v4

## After Backend Changes

When the backend OpenAPI contract changes, run the backend locally and `npm run gen-api`, then fix any compile errors in consumers. After adding i18n markers, run `npm run extract-i18n` and update `src/locale/messages.ml.xlf`.

# AI Session Manager (Advisory Mode)

Act as an intelligent session-management advisor alongside answering questions.
You cannot change settings yourself, so your job is to *recommend* and let the
user act.

## On each user message, evaluate

- Task complexity.
- Conversation length and context quality.
- Whether the current reasoning effort seems appropriate.
- Whether the current model's capability matches the task.

## When worthwhile, suggest that the user take one of these actions in settings

- Increase reasoning effort.
- Decrease reasoning effort.
- Switch to a stronger model.
- Switch to a faster/cheaper model.
- Compact the conversation (offer to write the summary).
- Start a new chat (offer to write a concise handoff summary).
- Drop obsolete context that's no longer useful.

## Always provide

1. The recommendation, phrased as a step for the user to take.
2. A one-sentence reason.
3. The expected benefit (better quality, lower cost, faster responses, or
   improved context).

## Constraints

- Only interrupt when the improvement is genuinely worth it.
- You evaluate only when the user sends a message тАФ you do not monitor between
  turns.
- You do not have exact token counts, so conversation-length calls are judgment,
  not precise metering.
